# 中台接入层骨架补全 — 设计规格

> **项目名称：** 医技影像平台  
> **设计日期：** 2026-04-25  
> **阶段定位：** 一期补缺

---

## 1. 背景与目标

当前仓库已经具备一期核心版的前后端主体能力，但中台接入层仍然偏向 LIS 单点实现：

- `DataSourceAdapter` 已定义统一接口
- `LISAdapter` 已提供一个现有实现
- `DataSyncScheduler` 已承担同步入口

设计规格与实现计划中提到的 RIS / PACS / EMR 接入、标准化流程和更清晰的同步编排边界尚未补齐。

本次目标不是接入真实外部系统，而是将中台接入层补成一个**可启动、结构完整、职责清晰**的后端骨架，为后续真实联调留出稳定边界。

---

## 2. 本次范围

### 2.1 必做

- 补齐接入适配器骨架：
  - `RISAdapter`
  - `PACSAdapter`
  - `EMRAdapter`
- 补齐标准化层骨架，使“适配器取数”和“平台内部模型转换”解耦
- 补齐同步编排层，使调度入口不再直接承担全部同步细节
- 保持 Spring 容器可以正常装配、启动和执行占位同步流程
- 让同步过程具备最小可观测性：source、状态、错误消息、时间等

### 2.2 明确不做

- 真实 LIS / RIS / PACS / EMR 外部系统联调
- 完整同步任务中心、任务队列、重试平台
- 前端管理页或系统配置页
- 大规模数据落表策略优化
- 对现有患者360、时间轴、检验详情、影像详情做功能扩展

---

## 3. 设计原则

- **延续现有分层。** 不推翻当前 `integration/adapter`、`integration/mapper`、`integration/sync` 的组织方式。
- **边界单一。** 适配器只负责来源系统访问；标准化服务只负责转换；编排服务只负责串联流程。
- **先稳定接口，再接真实系统。** 本次优先固定骨架和签名，不为了演示效果引入半成品联调逻辑。
- **失败可见。** 同步失败必须留下日志和状态，不做静默吞掉。
- **行为最小。** 在“能启动、能装配、能走通占位流程”的前提下，避免提前设计复杂扩展机制。

---

## 4. 目标结构

### 4.1 适配器层

保留 `DataSourceAdapter` 作为统一适配接口，并新增：

- `RISAdapter`：面向影像检查申请等 RIS 数据
- `PACSAdapter`：面向影像报告或影像系统引用信息
- `EMRAdapter`：面向患者、就诊等主数据

这些适配器在本次版本中都应满足：

- 作为 Spring Bean 可注入
- 能提供明确的 `sourceType`
- 能返回占位原始数据或空结果
- 能在日志中标识自己的来源系统

### 4.2 标准化层

新增统一标准化服务，用于把外部系统原始结果转为平台内部统一中间结构。

该层的职责：

- 统一命名和字段形态
- 将来源系统差异隔离在标准化层内
- 为后续真实映射和字段规则扩展提供固定入口

该层不直接负责：

- HTTP 调用
- 定时调度
- 业务表持久化策略

### 4.3 同步编排层

在 `integration/sync` 下新增同步编排服务，职责是：

1. 选择目标适配器
2. 调用适配器拉取占位数据
3. 调用标准化服务转换
4. 记录同步日志
5. 返回本次同步结果

`DataSyncScheduler` 继续作为调度入口保留，但应将具体同步流程下沉到编排层，不再把转换、保存、日志全部压在调度类里。

---

## 5. 数据流设计

本次统一同步链路如下：

`DataSyncScheduler -> SyncOrchestrator -> Adapter -> Standardizer -> SyncLog`

### 5.1 调度入口

- 定时任务触发同步
- 可以按 source 类型逐个执行
- 不负责具体字段映射和来源系统差异处理

### 5.2 适配器取数

- 适配器根据 `lastSyncTime` 或占位参数返回原始数据
- 本次允许返回空列表或静态占位数据
- 目标是保证签名与后续真实接入保持一致

### 5.3 标准化转换

- 将 RIS / PACS / EMR 原始对象转成统一中间结构
- 中间结构只服务于同步骨架，不要求一步到位覆盖所有业务实体
- 若某类数据当前只做占位，可返回最小必需字段

### 5.4 同步结果记录

- 每个 source 至少记录一次同步状态
- 记录内容包括：
  - sourceType
  - dataType 或 syncScope
  - status
  - errorMessage
  - createdAt / lastSyncTime

不要求本次实现完整统计报表，但要为后续扩展留出稳定字段。

---

## 6. 错误处理与可观测性

- 单个 source 同步失败时，记录失败日志并继续处理其他 source
- 同步编排层负责捕获可预期的接入错误并写入 `DataSyncLog`
- 调度入口不因一个占位源失败而整体中断
- 错误信息保留来源系统、同步阶段和异常消息

本次不新增复杂告警系统，但日志和同步记录必须足以判断：

- 哪个 source 失败
- 失败发生在什么阶段
- 是否影响后续 source 执行

---

## 7. 代码改动边界

本次设计预期主要涉及后端以下区域：

- `backend/src/main/java/com/medical360/integration/adapter/`
- `backend/src/main/java/com/medical360/integration/mapper/`
- `backend/src/main/java/com/medical360/integration/sync/`
- `backend/src/main/resources/application.yml`
- 与同步日志相关的最小实体 / mapper / 配置补充

允许对现有 `DataSyncScheduler` 做重构式调整，但应保持外部行为仍是“定时触发数据同步”。

---

## 8. 验收标准

本次完成后，应满足以下条件：

1. Spring Boot 应用可以正常启动
2. RIS / PACS / EMR 适配器可以作为 Bean 注入
3. 调度入口能够调用同步编排层
4. 至少一条占位同步链路可以走完“取数 -> 标准化 -> 记录状态”
5. `DataSyncLog` 能反映成功或失败结果
6. 代码结构能清楚回答三件事：
   - 谁负责访问外部系统
   - 谁负责标准化
   - 谁负责同步流程编排

---

## 9. 风险与控制

### 风险 1：把骨架做成半成品业务实现

如果在本次引入过多真实业务写表逻辑，范围会迅速扩大。

**控制方式：** 坚持“骨架优先”，只保留最小可运行路径，不承诺完整业务落表。

### 风险 2：职责继续堆在 `DataSyncScheduler`

如果只是继续往调度类里加代码，中台层会更难扩展。

**控制方式：** 把同步主流程下沉到编排服务，Scheduler 只保留触发职责。

### 风险 3：标准化层名义存在、实际无边界

如果适配器仍然自己做字段转换，后续真实接入时会反复改动。

**控制方式：** 明确要求适配器输出原始数据，标准化服务负责转换入口。

---

## 10. 后续演进

本次完成后，后续可以按以下顺序继续演进：

1. 为单个 source 补真实配置与联调
2. 引入更明确的中间模型和字段映射规则
3. 增加同步统计、重试和人工干预能力
4. 将标准化后的数据更稳定地写入业务表
5. 再补前端系统管理或接入监控页面

本设计故意把这些内容留在下一阶段，不在本次交付内。
